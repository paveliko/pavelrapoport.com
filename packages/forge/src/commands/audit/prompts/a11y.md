# CHECKLIST

Evaluate each item as PASS or FAIL with evidence:

1. **ARIA roles & attributes**: Interactive elements have correct roles, states, and properties. No redundant ARIA (e.g., `role="button"` on `<button>`, `role="link"` on `<a>`). Custom widgets use appropriate ARIA patterns (combobox, dialog, tablist, etc.).
2. **Keyboard navigation**: All interactive elements are focusable via Tab. Logical tab order follows visual layout. No keyboard traps — user can always Tab/Shift+Tab away. Escape closes modals, drawers, and popovers.
3. **Focus management**: Focus moves to modal/drawer/dialog on open and returns to trigger on close. Wizard and multi-step flows move focus to the new step. Skip-to-content link is present on pages with navigation.
4. **Color contrast**: Normal text meets 4.5:1 contrast ratio. Large text (18px+ or 14px+ bold) meets 3:1. UI components and graphical objects meet 3:1 against adjacent colors. No information conveyed by color alone — always paired with text, icon, or pattern.
5. **Form accessibility**: All inputs have visible, associated `<label>` elements (not just placeholder text). Error messages linked to inputs via `aria-describedby`. Required fields indicated with both visual marker and `aria-required="true"`. Form submission errors announce to screen readers.
6. **Image alt texts**: All `<img>` elements have meaningful `alt` text describing content, or `alt=""` for purely decorative images. Icon-only buttons and links have `aria-label` or visually hidden text. SVG icons used inline have `aria-hidden="true"` when decorative.
7. **Screen reader support**: Dynamic content updates use `aria-live` regions (toasts, loading states, inline validation). Heading hierarchy is logical (h1 > h2 > h3, no skipped levels). Page has landmark regions (`<main>`, `<nav>`, `<header>`, `<footer>` or ARIA equivalents).
8. **Touch targets**: All interactive elements are at least 44x44px on mobile viewports. Adequate spacing between adjacent touch targets to prevent mis-taps. Small inline links in text blocks have sufficient padding or touch area.
9. **Reduced motion**: `prefers-reduced-motion` media query respected — animations disabled or simplified when user prefers reduced motion. No auto-playing animations without user control. Framer Motion components use `useReducedMotion()` hook.
10. **RTL + a11y intersection**: Logical CSS properties used (`ms-`/`me-`, `ps-`/`pe-`, `start`/`end`, not `ml-`/`mr-`/`left`/`right`). Direction-aware `aria-label` values make sense in Hebrew context. Hebrew content sections have `lang="he"` attribute. Reading order in DOM matches visual RTL order.

## False positive to AVOID

Do NOT flag Radix UI or shadcn/ui primitives (Dialog, Select, Popover, etc.) for missing ARIA — these components handle accessibility internally. Also do not flag `aria-hidden="true"` on decorative SVG icons — this is correct usage. Do not flag physical CSS properties (`left-0`, `right-0`) in absolute/fixed positioning contexts where physical placement is intentional.
